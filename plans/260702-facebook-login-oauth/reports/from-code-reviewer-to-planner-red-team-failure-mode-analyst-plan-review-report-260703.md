## Finding 1: e2e `migrate deploy` + truncate can target the DEV database → dev data wipe; dev/test Postgres both bind 5432
- **Severity:** Critical
- **Location:** Phase 4, "Implementation Steps" step 3 + Phase 1, "Architecture" (docker-compose port map)
- **Flaw:** Phase 1 declares the schema datasource reads `env("DATABASE_URL")` (`phase-01:53,64`) and maps Postgres `5432:5432` (`phase-01:48`). Phase 4 introduces a *second* variable name `DATABASE_URL_TEST` in `.env.test` (`phase-04:29,35`) and says `migrate deploy ... nhắm vào DATABASE_URL_TEST` plus "Reset data giữa các test bằng truncate". But Prisma's datasource only ever reads `DATABASE_URL` — it does not read `DATABASE_URL_TEST`. The plan specifies **no wiring** (no `dotenv-cli`, no `cross-env DATABASE_URL=...`, no second datasource) that makes `migrate deploy`/the e2e Prisma client actually connect to the test URL. `grep -rn "dotenv-cli\|cross-env" plans/...` → no match.
- **Failure scenario:** Engineer follows Phase 4 literally: `.env` still exports `DATABASE_URL` (dev), `.env.test` exports `DATABASE_URL_TEST`. The e2e harness boots with `DATABASE_URL` still pointing at the **dev** DB (the only var Prisma honors), then runs the between-test **`TRUNCATE`** — wiping the developer's local dev data every test run. Simultaneously, if `docker-compose.test.yml` copies the dev port map `5432:5432` while the dev Postgres is up, the test container fails to bind the port and the whole e2e suite can't start (or silently reuses the dev container). Data loss is unrecoverable (no dump step specified).
- **Evidence:** `phase-01-schema-prisma-user.md:48` (`port map 5432:5432`), `phase-01:53,64` (`DATABASE_URL`), `phase-04-tests.md:29,35` (`DATABASE_URL_TEST` + "migrate deploy ... nhắm vào DATABASE_URL_TEST" + "truncate"); no override mechanism anywhere in plan (grep empty).
- **Suggested fix:** Make the test path physically incapable of hitting dev: (a) run e2e via `dotenv -e .env.test -- prisma migrate deploy` / `dotenv -e .env.test -- jest --config test/jest-e2e.json`, i.e. load `.env.test` so `DATABASE_URL` itself is the test URL at runtime (drop the `DATABASE_URL_TEST` name — Prisma never reads it); (b) map the test container to a **different host port** (e.g. `5433:5432`) so it can coexist with dev; (c) add a guard in the e2e truncate helper that asserts the connected DB name contains `test` before truncating.

## Finding 2: JWKS network/timeout failures are reported to clients as `401 Invalid Facebook token` — server outage blamed on the client, and total-login-outage recovery gap
- **Severity:** High
- **Location:** Phase 2, "Architecture" (catch-all) + "Risk Assessment"
- **Flaw:** `createRemoteJWKSet(...)` performs a live network fetch of Facebook's JWKS on first verify (and on `kid` cache-miss/rotation). Phase 2 catches **every** error from `jwtVerify` — including `jose` infra errors `JWKSTimeout`, `JWKSNoMatchingKey`, and the underlying `fetch` failure — and collapses them all into `UnauthorizedException('Invalid Facebook token')` (`phase-02:45,62`). The Risk Assessment's own mitigation says "log rõ lỗi network riêng biệt với lỗi verify token" (`phase-02:76-77`), but the architecture code does the opposite: one catch, one 401, no discrimination.
- **Failure scenario:** Facebook's JWKS endpoint is slow or unreachable (DNS blip, egress firewall, Meta incident, or just cold-start on the very first login after deploy). `jose` throws `JWKSTimeout`. The service returns `401 Invalid Facebook token` to every user with a perfectly valid token. On-call sees "auth failures," users see "your login is invalid," and the true cause (outbound network to Facebook) is invisible because it was laundered into a 401. There is no retry, no circuit-breaker, no cached-last-known-good-JWKS fallback — a Facebook JWKS outage is a total login outage with a misleading signal.
- **Evidence:** `phase-02-facebook-verify-service.md:29` (module-level `createRemoteJWKSet` = network dependency), `:45` ("catch chung, ném `UnauthorizedException`"), `:62` (step 4 catches "mọi lỗi từ `jwtVerify`"), `:76-77` (mitigation claims separate network logging that the code doesn't implement).
- **Suggested fix:** Distinguish infra errors from token errors: catch `jose` errors and if the error is a JWKS/network class (`JWKSTimeout`, `JWKSNoMatchingKey`, fetch/`ECONNREFUSED`/`ETIMEDOUT`), throw `ServiceUnavailableException` (503) and log at error level; only true claim/signature failures (`JWTExpired`, `JWTClaimValidationFailed`, `JWSSignatureVerificationFailed`) become 401. Set explicit `timeoutDuration`/`cooldownDuration`/`cacheMaxAge` on `createRemoteJWKSet` rather than relying on "defaults hợp lý."

## Finding 3: No env validation → if `FACEBOOK_APP_ID` is missing/mistyped in any environment, jose SILENTLY skips `aud` validation → tokens from ANY Facebook app are accepted
- **Severity:** High
- **Location:** Phase 2, "Architecture" (`audience: FACEBOOK_APP_ID`) — plan-wide absence of config validation
- **Flaw:** Verification passes `audience: FACEBOOK_APP_ID` sourced from `ConfigService` (`phase-02:34,47`). Confirmed from jose source (`jwt_claims_set.ts`): audience is only checked `if (audience && ...)` and `'aud'` is only added to required-claims `if (audience !== undefined)`. So when `FACEBOOK_APP_ID` resolves to `undefined`/empty (env not loaded in a given environment, typo, forgotten in staging/prod `.env`), jose **does not validate `aud` at all** — signature + issuer still pass, but any token minted for *any other* Facebook app is accepted. The plan adds `@nestjs/config` but specifies no `validationSchema`/fail-fast (`grep -rn "validationSchema" plans/...` → no match).
- **Failure scenario:** Ops deploys to a new environment and forgets `FACEBOOK_APP_ID` (easy — `.env.example` ships it blank: `my-expo-app/.env.example:3` is `FACEBOOK_APP_ID=`). App boots fine (no validation). Login "works." An attacker presents a valid Facebook Limited-Login JWT issued for a *different* app they control → passes issuer + signature, `aud` unchecked → they are minted a system JWT for a `User` created from their `sub`. The audience pin — the one thing separating "a real Facebook token" from "a token for our app" — is silently disabled by a missing env var. Same class of late failure applies to a missing `JWT_SECRET` / `DATABASE_URL`.
- **Evidence:** `phase-02:34,47` (`audience: FACEBOOK_APP_ID` from ConfigService); jose source `jwt_claims_set.ts` confirms aud check gated on `if (audience && ...)` / required-claim push gated on `if (audience !== undefined)`; no `validationSchema` in plan (grep empty); `my-expo-app/.env.example:3` blank value shows the "forgot to fill" path is realistic.
- **Suggested fix:** Add a Joi/zod `validationSchema` to `ConfigModule.forRoot({ validationSchema, validationOptions: { allowUnknown: true, abortEarly: false } })` requiring `FACEBOOK_APP_ID`, `JWT_SECRET`, `DATABASE_URL` as non-empty — fail fast at boot. Defense-in-depth: assert `FACEBOOK_APP_ID` is truthy before `jwtVerify` and hard-fail if not.

## Finding 4: `JwtModule.register({ secret: process.env.JWT_SECRET })` reads env at import time — before `@nestjs/config` loads `.env` → `secret` is `undefined` at boot
- **Severity:** High
- **Location:** Phase 3, "Implementation Steps" step 2 (`JwtModule.register`)
- **Flaw:** Phase 3 registers `JwtModule.register({ secret: process.env.JWT_SECRET, signOptions: { expiresIn: '7d' } })` (`phase-03:33,123`) — the **synchronous** form. This evaluates `process.env.JWT_SECRET` at module-metadata-construction time, which runs when `auth.module.ts` is first `import`ed. `@nestjs/config`'s `ConfigModule.forRoot()` (the thing that runs dotenv and populates `process.env` from `.env`) executes *later*, when `AppModule`'s imports are instantiated. There is no `.env` auto-loading before that. So unless `JWT_SECRET` is exported in the actual shell env, `process.env.JWT_SECRET` is `undefined` at the moment `JwtModule.register` captures it.
- **Failure scenario:** Developer sets `JWT_SECRET` only in `.env` (as the plan says: `phase-03:117`) and starts the app. `ConfigModule` loads it into `process.env` at runtime, but `JwtModule` already froze `secret: undefined` during import. `jwtService.sign(...)` then throws `Error: secretOrPrivateKey must have a value` → **every** `POST /auth/facebook` and `/link-confirm` returns 500. Nothing in the plan's Success Criteria catches this (the unit tests mock/inject, the e2e overrides the verify provider but still needs real signing). Classic NestJS footgun.
- **Evidence:** `phase-03-auth-endpoint-va-account-linking.md:33` (comment) and `:123` (step 2) both use `JwtModule.register({ secret: process.env.JWT_SECRET })`; Phase 2 introduces `@nestjs/config` (`phase-02:47,60`) as the env loader; `grep -rn "registerAsync\|useFactory" plans/...` → no match (async form never specified).
- **Suggested fix:** Use `JwtModule.registerAsync({ inject: [ConfigService], useFactory: (c) => ({ secret: c.getOrThrow('JWT_SECRET'), signOptions: { expiresIn: '7d' } }) })`. `getOrThrow` also gives fail-fast for the missing-secret case in Finding 3.

## Finding 5: Docker Compose readiness race — `migrate` and `PrismaService.onModuleInit` connect before Postgres accepts connections; no healthcheck, no retry
- **Severity:** High
- **Location:** Phase 1, "Implementation Steps" steps 3+6 + "Architecture" (`onModuleInit`); Phase 4 step 3 (test compose)
- **Flaw:** Phase 1 step 3 runs `docker compose up -d` then step 6 immediately runs `npx prisma migrate dev --name init` (`phase-01:63,66`). `docker compose up -d` returns as soon as the container *starts*, not when Postgres is *ready to accept connections* (several seconds later on first boot: initdb + WAL). No `healthcheck`, no `depends_on: condition: service_healthy`, no wait-for-postgres is specified (`grep -rni "healthcheck\|depends_on\|wait" plans/...` → no match). Same for the automated Phase 4 flow (`docker-compose.test.yml` → `migrate deploy`, `phase-04:35`). And `PrismaService.onModuleInit` does a bare `await this.$connect()` (`phase-01:44`) with no retry.
- **Failure scenario:** (a) Fresh clone, `docker compose up -d && npx prisma migrate dev` → `migrate` hits `ECONNREFUSED`/`the database system is starting up` because Postgres isn't ready → migration aborts, first-run appears "broken." (b) In CI/e2e the same race is nondeterministic → flaky red builds. (c) If anyone ever runs the app in the same compose without a health-gated `depends_on`, `onModuleInit`'s un-retried `$connect()` throws → Nest bootstrap fails → container exits (crash loop), no recovery.
- **Evidence:** `phase-01:63` (`docker compose up -d`), `:66` (`migrate dev` immediately after), `:44` (`await this.$connect()` no retry), `phase-04:35` (test compose → `migrate deploy`); no healthcheck/wait anywhere (grep empty).
- **Suggested fix:** Add a `healthcheck` (`pg_isready`) to both compose files and gate dependents on `condition: service_healthy`; in the migrate/test scripts, wait-for-ready before `migrate`; optionally wrap `$connect()` in a bounded retry. State the ordering explicitly in the steps rather than "up -d then immediately migrate."

## Finding 7: [PERSISTS — restructure did not fix] P2002 `email`-branch refetch returns the email-matched account straight into `signToken`, reopening the pre-account-takeover the plan claims to prevent
- **Severity:** Critical
- **Location:** Phase 3, "Architecture" — `findOrCreateFacebookUser` P2002 catch (`phase-03:71`), consumed by JWT issue (`phase-03:32-33,48`)
- **Flaw (evaluating the plan's own no-transaction race handling, per assignment):** The no-transaction decision *itself* is sound — separate request connections can't be serialized into not-inserting; the `@unique` index is the correct duplicate-preventer, and catch-P2002-refetch is a legitimate pattern. The failure mode is the **refetch of the `email` branch**: `if (target.includes('email')) return prisma.user.findUniqueOrThrow({ where: { email: profile.email! } })` (`phase-03:71`) returns that row directly, and the controller turns it into a session `signToken(user.id)` (`phase-03:32-33`). The non-race path throws `409 EMAIL_LINK_REQUIRED` for exactly this email-match case (`phase-03:56-60`); the race path silently skips that gate.
- **Failure scenario:** TOCTOU. `findUnique({email})` at `phase-03:57` returns null; a concurrent write commits that email in the window; the `create` throws P2002-on-`email`; the catch refetches and returns the *other* account (a password account, or a different Facebook identity sharing the email) → the caller is minted a valid system JWT for an account they never proved ownership of. This is the precise "silent auto-link / pre-account hijacking" the accepted Option-B decision exists to block — reintroduced through the concurrency fix. This defect was flagged Critical in the 2026-07-02 pass (security-adversary report Finding 1, failure-mode report Finding 1) and is **still present verbatim** in the current phase text.
- **Evidence:** `phase-03-auth-endpoint-va-account-linking.md:71` (`return ... findUniqueOrThrow({ where: { email }})`), `:56-60` (the 409 gate it bypasses), `:32-33,48` (`signToken` on the returned user); unchanged from prior reports (`reports/from-code-reviewer-to-planner-red-team-security-adversary-plan-review-report.md:27-33`).
- **Suggested fix:** In the `email`-target branch, do NOT return the row — re-apply the branch decision: if the refetched row has a `passwordHash` or a `facebookId` that differs from `profile.facebookId`, throw `409 EMAIL_LINK_REQUIRED`. Only refetch-and-return in the `facebookId`-target branch (true same-identity race). Add a Phase 4 e2e that races two requests with *different facebookId + same email* and asserts 409, never a 200+JWT (current e2e only races *same* facebookId — `phase-04:42` — so this branch has zero coverage).

## Finding 8: [PERSISTS] P2002 recovery is connector-fragile, and `link-confirm` has two unhandled-500 paths (null email, already-linked-elsewhere)
- **Severity:** Medium
- **Location:** Phase 3 — `findOrCreateFacebookUser` (`phase-03:69-71`) and `linkFacebookAccount` (`phase-03:90-108`)
- **Flaw:** (a) `const target = (e.meta?.target as string[]) ?? []` then `target.includes('facebookId')` (`phase-03:69-71`). Prisma's `meta.target` shape is connector-dependent — on Postgres it has historically been the constraint-name **string** (e.g. `"User_email_key"`), not `['email']`. `.includes()` on a string is a substring match, so the branch dispatch only works *by coincidence* and the `as string[]` cast is false. No Prisma version is pinned (`package.json` has no prisma dep — verified). (b) `linkFacebookAccount` does `findUnique({ where: { email: profile.email } })` (`phase-03:90`) but `profile.email` can be `null` (`phase-02:41` — email permission denied is a supported path); `where: { email: null }` throws `PrismaClientValidationError` (not P2002, caught by nothing) → 500. (c) `linkFacebookAccount` never checks whether `profile.facebookId` is already linked to a *different* user before `update({ facebookId })` (`phase-03:104-107`) → collides with a pre-existing `facebookId` → P2002 → no catch → 500 + identity ambiguity.
- **Failure scenario:** (a) After a future Prisma upgrade during real implementation, `meta.target` format shifts → `.includes('facebookId')` is false → the genuine race falls through to `throw e` → the exact 500 the catch was written to prevent. (b) A Facebook-only user (email denied) hits `/auth/facebook/link-confirm` → `findUnique({email:null})` → 500 on a public auth endpoint. (c) FB identity F already on user U2; U1 (password, same email) does link-confirm with F's token + U1 password → `update({facebookId:F})` → P2002 → 500.
- **Evidence:** `phase-03:69-71` (`as string[]` + `.includes`), `phase-03:90` (`findUnique({email: profile.email})`) vs `phase-02:41` (`email: ... ?? null`), `phase-03:104-107` (unguarded `update` on unique `facebookId`, no prior facebookId lookup). Same as prior failure-mode report Findings 2–4; still present.
- **Suggested fix:** Normalize `meta.target` to handle both string and string[] (or match on `e.code==='P2002'` + re-run both lookups rather than trusting target contents); guard `profile.email == null` before the email `findUnique` in `linkFacebookAccount` (throw a clean 400/404); add a `findUnique({facebookId})` already-linked-elsewhere check + wrap the `update` in the same P2002 catch. Add Phase 4 coverage for all three.
